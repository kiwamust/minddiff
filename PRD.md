> **Status:** Draft

---

## 1. Executive Summary

チームの認識ギャップを可視化するGenAIツール。

各メンバーが持つメンタルモデル（目的・状況・リスク・合意・優先順位）を定期的に収集し、LLMが統合・Diff生成することで、直接対話なしに乖離を検出する。

**一行仮説：** 「認識合わせに必要なコミュニケーション量」が「実行可能なコミュニケーション量」を超えている時、その差分をAgentが埋められるか。

---

## 2. Problem Statement

### 現状の痛み

プロジェクトチームにおける認識ズレは、以下の経路で損失を生む：

1. **断片的なSlack** — 情報は流れるが、蓄積・統合されない
2. **たまの定例** — 頻度が足りない。全員が同じ解像度で語れない
3. **更新されない議事録** — 合意事項が形骸化する
4. **暗黙の前提の不一致** — 「完成」「十分」「リスク」の定義が人によって異なる

これらの乖離はプロジェクト後半にコストの高い手戻りとして顕在化する。

### なぜ今か

LLMが「複数の自然言語テキストの意味的Diff」を実用レベルで生成できるようになった。これは2年前には不可能だった能力。技術的タイミングが来ている。

---

## 3. Core Hypothesis

> **各メンバーのメンタルモデルをAgentが収集・統合し、乖離を可視化することで、直接対話なしに認識ギャップを検出・解消できるか。**

検証すべき2軸：

| 軸       | 問い                                                        |
| -------- | ----------------------------------------------------------- |
| **精度** | Agentが生成する統合と乖離検出は、実態を正しく反映しているか |
| **効率** | Agentを介した認識合わせは、同等の対話より総コストが低いか   |

---

## 4. Design Principles

### 4.1 最重要原則：偽の収束を防ぐ

Agentがもたらしうる最悪の結果は、**実際には乖離が存在するのに「全員一致しています」と報告すること**。

チームが乖離を認識していれば、それは議論で解決できる。偽の収束はその機会を奪い、手戻りコストを増幅する。

> **収束を断定するより、乖離を過検出する方が安全である。**

Agentは「ここは一致しています」と言うことには慎重であるべきで、「ここにズレがある可能性があります」と言うことには積極的であるべきである。

### 4.2 判断しない

Agentは「Aが正しく、Bが間違っている」とは言わない。乖離の存在を提示するだけ。解消はチームが行う。

### 4.3 入力過程そのものに価値がある

「今、自分は何を理解しているか？」を言語化する行為は、メタ認知を強制する。多くの認識ズレは、メンバーが自分の認識を言語化していないことに起因する。ツールへの入力行為がすでに介入である。

### 4.4 正しいベースラインで比較する

「コミュニケーションゼロ」と比較すれば当然勝つ。「無制限の対面対話」と比較すれば当然負ける。正しいベースラインは**「現実の制約下での既存コミュニケーション」**。断片的なSlack、たまの定例、更新されない議事録、という現状との比較。

## 5. Product Concept

### コンセプト名

**メンタルモデルDiffツール** — チームのメンタルモデルに対する `git diff` を生成する。

### メンタルモデルの5次元

| #   | 次元       | 問い                                                                                            | 設計意図                                                             | 検出対象         |
| --- | ---------- | ----------------------------------------------------------------------------------------------- | -------------------------------------------------------------------- | ---------------- |
| 1   | 目的理解   | この1〜2週間で、チームとして最も達成すべきことは何ですか？ それはなぜですか？                   | 時間アンカー。「なぜ」で目的レベルの認識差を引き出す                 | 目的の乖離       |
| 2   | 状況認識   | 今、最も順調に進んでいることと、最も遅れている・詰まっていることは何ですか？                    | 二項構造で粒度を揃える。「順調」と「遅れ」の認識差は高シグナル       | 状況認識の乖離   |
| 3   | リスク認知 | 今、最も心配していることを2〜3つ挙げてください。                                                | 複数回答で網羅性を上げる。「心配」は「リスク」より率直な言語化を促す | リスク認知の乖離 |
| 4   | 合意事項   | 過去1〜2週間で、チームとして決まった（と自分が認識している）ことを挙げてください。              | 時間アンカー。「と自分が認識している」で認識の主観性を明示的に問う   | 合意事項の乖離   |
| 5   | 優先順位   | 来週、自分が最も注力すべきことは何ですか？ また、チーム全体として最も注力すべきことは何ですか？ | 個人とチームを分離。この2つのズレ自体が乖離シグナル                  | 優先順位の乖離   |

**問い設計の原則：** Diffable（比較可能な粒度に揃う）、Time-anchored（時間範囲を明示）、Signal-rich（解釈・判断を引き出す）、Low-friction（5問合計5分以内）。MVP1では固定。文言の妥当性はPoC後に評価しMVP2で調整可能にする。

## 6. User Stories

### プライマリペルソナ

**PM / テックリード**（5〜10名チームのまとめ役）

- 全員の認識を揃える責任があるが、1on1を全員分回す時間がない
- 定例で「認識合ってるよね？」と聞いても、ズレは表面化しない

**チームメンバー**（開発者 / デザイナー / QA）

- 自分の理解が全体と合っているか不安だが、確認する場が少ない
- 他メンバーが何をリスクと捉えているか知らない

### ストーリー

| ID    | As a...  | I want to...                             | So that...                                           |
| ----- | -------- | ---------------------------------------- | ---------------------------------------------------- |
| US-01 | PM       | 全メンバーの目的理解を一覧で見たい       | 認識ズレを定例前に把握し、議論を焦点化できる         |
| US-02 | PM       | メンバー間のリスク認知の差を可視化したい | 見落とされているリスクを早期に検出できる             |
| US-03 | PM       | 「合意事項」の認識差を検出したい         | 決まったはずのことが実は共有されていない状況を防げる |
| US-04 | メンバー | 週1回、5つの問いに答えるだけで済ませたい | 認識合わせのために追加の会議に出なくて済む           |
| US-05 | メンバー | チーム全体の認識マップを閲覧したい       | 自分の理解がチームと合っているか確認できる           |
| US-06 | メンバー | 乖離レポートで自分の認識のズレを知りたい | 対話が必要な論点を事前に準備できる                   |
| US-07 | PM       | 乖離度の推移を時系列で見たい             | 認識合わせの効果をトラッキングできる                 |

## 9. Architecture

### システム構成

```
┌─────────────┐     ┌──────────────┐     ┌──────────┐
│   Web UI    │────▶│  API Server  │────▶│   LLM    │
│  (入力/閲覧) │◀────│  (集約/保存)  │◀────│  (統合)  │
└─────────────┘     └──────┬───────┘     └──────────┘
                           │
                    ┌──────▼───────┐
                    │   Simple DB  │
                    │ (入力/出力保存)│
                    └──────────────┘
```

**設計判断：** マイクロサービスでもマルチAgentでもメッセージキューでもない。**モノリスで十分。** PoCの目的はアーキテクチャの検証ではなく、仮説の検証。

### 技術スタック（候補）

| レイヤー | 選択肢                     | 判断基準                                                   |
| -------- | -------------------------- | ---------------------------------------------------------- |
| Frontend | Next.js / Remix            | SSR不要ならSPA + API でも可                                |
| Backend  | Python (FastAPI) / Node.js | LLMライブラリとの親和性                                    |
| DB       | PostgreSQL                 | RDBで十分。JSONBカラムで柔軟性確保                         |
| LLM      | Claude API / OpenAI API    | プロンプトエンジニアリングで対応。ファインチューニング不要 |
| Hosting  | Fly.io / Railway / Render  | PoCなので最小構成                                          |

### データモデル（最小）

```
Team
  - id, name, created_at

Member
  - id, team_id, name, email, role (pm | member)

InputCycle
  - id, team_id, cycle_number, start_date, end_date, status

Response
  - id, input_cycle_id, member_id, dimension (1-5), content, submitted_at

Report
  - id, input_cycle_id, type (synthesis | divergence), dimension, content, confidence, generated_at
```

## 10. Output Design

### チーム認識マップ

5次元それぞれについて、チーム全体の認識を統合した要約。

```
■ 目的理解
  「Q1リリースに向けた機能Xの実装」
  一致度: ██████░░░░ 60%

■ 状況認識
  「バックエンド実装完了、フロント50%、テスト未着手」
  一致度: ████████░░ 80%

■ リスク認知
  「外部API依存とスケジュール」
  一致度: ████░░░░░░ 40%  ⚠

■ 合意事項
  「APIスキーマはv2で確定、デプロイはCI/CD経由」
  一致度: ███████░░░ 70%

■ 優先順位
  「フロント完成 → テスト → ドキュメント」
  一致度: █████░░░░░ 50%  ⚠
```

### 乖離レポート

各次元について、メンバー間の見解が分かれている箇所を具体的に提示。

```
■ 目的理解
  ⚠ 乖離検出 [確信度: 高]
    概念: 「完成」の定義にズレがある
    根拠:
      - メンバー①:「最低限の動作確認ができればOK」
      - メンバー③:「本番品質まで仕上げる前提で動いている」
    💡 推奨アクション: Doneの定義を明示的に合意する

■ リスク認知
  ⚠ 乖離検出 [確信度: 中]
    概念: 外部API依存リスクがチーム全体で共有されていない
    根拠:
      - メンバー②:「外部APIの仕様変更が最も心配」
      - 他メンバーの入力にこのリスクへの言及なし（0/4名）
    💡 推奨アクション: リスク一覧をチームで共有する

■ 合意事項
  ⚠ 乖離検出 [確信度: 低]
    概念: 「APIスキーマv2確定」の合意が全員に届いていない可能性
    根拠:
      - メンバー①②④: 合意事項として言及あり
      - メンバー③: 言及なし（認識漏れ or 単に省略）
    💡 推奨アクション: 確認が必要（偽陽性の可能性あり）
```

**レポート構造の原則:** 概念レベルで乖離を検出し、文レベルの引用で裏付ける二層構造。メンバーは匿名番号で表示。

**重要：** Agentは「Aが正しく、Bが間違っている」とは言わない。乖離の存在と推奨アクションを提示するだけ。解消はチームが行う。

## 11. PoC Plan

### 対象

以下を満たす実プロジェクトチーム：

- 5〜10名規模
- プロジェクト進行中（キックオフ直後やクロージング中でない）
- 定例はあるが、認識ズレが時々顕在化するレベル（極端でない中間地帯）
- PMがツールの意義を理解し、運用を主導できる

### 期間

2〜3スプリント（4〜6週間）。

### プロトコル

| Phase                | 期間     | 内容                                                                                                                                |
| -------------------- | -------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| **ベースライン測定** | Week 1-2 | ツール導入前に5次元の認識調査を実施。個別インタビューで「実際の乖離」を人手で特定。認識合わせに費やしている時間を計測               |
| **ツール導入**       | Week 3-4 | 週1〜2回、各メンバーがツールに入力。AI生成の認識マップと乖離レポートを全員に共有。乖離ポイントの解消に使った時間を記録              |
| **効果測定**         | Week 5-6 | 乖離度の変化を測定。コミュニケーション総時間の変化を測定。偽の収束の有無を確認（Agent報告 vs 実態の突合）。メンバーの主観評価を収集 |

### 成功基準

| 指標         | 基準                                                                          | 測定方法                                |
| ------------ | ----------------------------------------------------------------------------- | --------------------------------------- |
| 乖離検出精度 | Agentが検出した乖離の80%以上が実際に存在する（偽陽性 ≤ 20%）                  | Agent報告 vs インタビューによる実態突合 |
| 偽収束率     | Agentが「一致」と判定したうち、実際に乖離があったケースが10%以下              | 同上                                    |
| 時間効率     | 認識合わせに費やす総時間（入力＋閲覧＋解消対話）がベースライン比で30%以上削減 | 時間ログ                                |
| 主観評価     | メンバーの70%以上が「有用」と回答                                             | アンケート                              |

## 12. Risks & Mitigations

| リスク                                                                            | 深刻度   | 軽減策                                                                                                                                                                               |
| --------------------------------------------------------------------------------- | -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **偽の収束** — 乖離があるのに「一致」と報告                                       | Critical | 設計原則として乖離の過検出を許容。一致判定の閾値を高く設定。PoC中にAgent報告と実態を突合して偽収束率を測定                                                                           |
| **入力の形骸化** — メンバーが適当に回答する                                       | High     | 入力負荷を最小化（5問×短文）。入力が認識整理に役立つという実感を早期に醸成                                                                                                           |
| **プライバシー懸念** — 率直な回答が躊躇される                                     | High     | 匿名化して全員閲覧可とし透明性を担保。ただし5〜10名チームでは文体から推測可能（脆い匿名性）。運用ルールで「誰が書いたかの詮索禁止」を明記。PoC中にこの脆さが入力品質に影響するか観察 |
| **LLMの幻覚** — 入力にない乖離を捏造する                                          | Medium   | プロンプトに「入力テキストに根拠がない乖離を生成しない」制約を明記。出力に根拠テキストの引用を含める                                                                                 |
| **情報の劣化** — Agentの統合で本来のニュアンスが失われる                          | Medium   | 統合ビューから元の入力テキストへのドリルダウンを可能にする。統合はあくまでサマリーであり、原文が正                                                                                   |
| **ベースライン測定の困難** — 「現状のコミュニケーション時間」の正確な計測が難しい | Medium   | 完全な精度は求めない。自己申告ベースの推定値で比較                                                                                                                                   |

## 13. Scope

### MVP1 In-Scope

- 5次元の手動入力フォーム
- LLMによる統合・Diff生成（バッチ処理）
- チーム認識マップ・乖離レポートの閲覧
- 基本的なチーム管理（作成・招待）
- 入力サイクル管理（週次）
- 全員同一ビュー（パーソナライズなし）

### MVP1 Out-of-Scope

| 項目                       | 理由                             | 対象MVP |
| -------------------------- | -------------------------------- | ------- |
| Slack / Jira等との自動連携 | 手動入力の価値を先に検証する     | MVP2    |
| リアルタイム処理           | バッチで十分。複雑さを避ける     | MVP2+   |
| パーソナライズビュー       | まず全員同一ビューで検証する     | MVP3    |
| 随伴Agent（常時伴走）      | MVP4の世界観。MVP1の結果が決める | MVP4    |
| モバイルアプリ             | レスポンシブWebで対応            | —       |
| SSO / SAML                 | PoC段階では不要                  | MVP2+   |

## 14. Evolution Path

```
MVP1: メンタルモデルDiff（手動入力、バッチ処理、全員同一ビュー）
  ↓ 仮説が検証されたら
MVP2: 入力の半自動化（Slack/Jira等からの情報自動収集）
  ↓
MVP3: パーソナライズ（メンバーの役割・関心に応じた情報フィルタリング）
  ↓
MVP4: 随伴Agent化（各メンバーに常時伴走するAgent）
```

当初の設計（TMA/PMA/EMA、マイクロサービス、イベントバス）はMVP4の世界観。そこに到達すべきかどうかは、MVP1の結果が決める。

## Appendix A: LLMプロンプト設計方針

MVP1はファインチューニング不要。プロンプトエンジニアリングで対応する。

### 統合プロンプトの設計原則

1. **入力テキストの忠実な統合** — 意味を変えない。抽象化しすぎない
2. **根拠の明示** — 統合見解がどのメンバーの入力に基づくか追跡可能にする
3. **乖離の過検出** — 「一致」判定の閾値を高く設定。微妙な表現の違いも乖離候補として報告
4. **捏造の抑制** — 入力テキストに根拠がない推論を生成しない
5. **確信度の付与** — 高/中/低で乖離の確からしさを段階評価

### プロンプト構造（概念）

```
[System] あなたはチームの認識ギャップを検出するアナリストです。
         以下の原則に従ってください：
         - 一致の判定には慎重に。乖離の検出には積極的に。
         - 入力にない情報を推測・創作しない。
         - 乖離には確信度（高/中/低）を付与する。

[Input]  5次元 × N名分の回答テキスト

[Task]   1. 各次元の統合見解を生成
         2. メンバー間の乖離を検出・レポート
         3. 推奨アクションを提案
```

## Appendix B: 非自明な洞察

MVP1を設計する過程で得られた、仕様書には書ききれない洞察。

1. **入力過程そのものが介入である** — ツールの価値はアウトプット（レポート）だけではない。「今、自分は何を理解しているか？」を言語化する行為がメタ認知を強制する。多くの認識ズレは、メンバーが自分の認識を言語化していないことに起因する

2. **正しい比較対象を選ぶ** — 「コミュニケーションゼロ」と比較すれば当然勝つ。「無制限の対面対話」と比較すれば当然負ける。正しいベースラインは「断片的なSlack、たまの定例、更新されない議事録」という現状

3. **偽の収束は偽の乖離より危険** — 偽陽性（存在しない乖離の報告）は無駄な議論を生むだけ。偽陰性（見逃された乖離）はプロジェクト後半の手戻りを生む。コストの非対称性がある

4. **このツールは「認識合わせの代替」ではなく「認識合わせの焦点化」** — 対話をゼロにするのではなく、「何について話すべきか」を明確にする。限られた対話時間を最も乖離が大きい論点に集中させる
